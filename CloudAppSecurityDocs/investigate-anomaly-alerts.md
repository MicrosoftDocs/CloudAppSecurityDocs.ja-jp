# <a name="how-to-investigate-anomaly-detection-alerts"></a>異常検出アラートを調査する方法

*適用対象:Microsoft Cloud App Security*

Microsoft Cloud App Security は、悪意のあるアクティビティのセキュリティ検出とアラートを提供します。 このガイドの目的は、調査と修復のタスクを支援するために、各アラートについての一般的で実用的な情報を提供することです。 このガイドでは、アラートをトリガーする条件に関する一般的な情報について説明します。 ただし、異常検出は本質的には不明確であるため、標準とは異なる動作が発生した場合にのみトリガーされることに注意してください。 最後に、一部のアラートがプレビュー段階にある可能性があるため、更新されたアラートの状態に関する公式ドキュメントを定期的に確認してください。

## <a name="mitre-attck"></a>MITRE ATT ・・ AT&T \&

Cloud App Security アラートと使い慣れた MITRE ATT のマトリックスとの間の関係を簡単にマップできるようにするために、 \& アラートを対応する MITRE att の手法で分類しました \& 。 この追加の参照により、Cloud App Security アラートがトリガーされたときに使用される可能性のある攻撃手法を理解しやすくなります。

このガイドでは、次のカテゴリの Cloud App Security アラートの調査と修復に関する情報を提供します。

## <a name="security-alert-classifications"></a>セキュリティ アラートの分類

適切な調査の後、すべての Cloud App Security アラートは、次のいずれかの種類のアクティビティとして分類できます。

- **真陽性 (TP)**: 悪意のあることが確認されたアクティビティに関するアラート。
- **害のない真陽性 (B TP)**: 侵入テストやその他の承認された疑わしいアクションなど、疑わしいが悪意のあるアクティビティに関するアラート。
- **偽陽性 (FP)**: 悪意のないアクティビティに関するアラート。

## <a name="general-investigation-steps"></a>一般的な調査手順

推奨される操作を適用する前に、潜在的な脅威をより明確に理解するために、任意の種類のアラートを調査するときは、次の一般的なガイドラインを使用する必要があります。

- ユーザーの調査の[優先度スコア](tutorial-ueba.md#understand-the-investigation-priority-score)を確認し、組織の他の部分と比較します。 これは、組織内のどのユーザーが最も大きなリスクをもたらすかを特定するのに役立ちます。
- **TP**を識別する場合は、すべてのユーザーのアクティビティを確認して、影響について理解を深めます。
- すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認し、ソースと影響の範囲を調べます。 たとえば、次のユーザーデバイス情報を確認し、既知のデバイス情報と比較します。
  - オペレーティング システムとバージョン
  - ブラウザーとバージョン
  - IP アドレスと場所

## <a name="initial-access-alerts"></a>初期アクセスアラート

このセクションでは、悪意のあるアクターが組織に最初の足掛かりを獲得しようとしている可能性があることを示すアラートについて説明します。

### <a name="activity-from-anonymous-ip-address"></a>匿名 IP アドレスからのアクティビティ

**説明**

Microsoft の脅威インテリジェンスまたは組織によって匿名のプロキシ IP アドレスとして識別された IP アドレスからのアクティビティ。 これらのプロキシを使用してデバイスの IP アドレスを非表示にし、悪意のあるアクティビティに使用することができます。

**Tp**、 **B tp**、または**FP**。

この検出では、機械学習アルゴリズムを使用して、組織内のユーザーが広く使用している mistagged IP アドレスなど、 **B TP**インシデントを減らします。

1. **TP**: アクティビティが匿名または TOR IP アドレスから実行されたことを確認できます。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **B TP**: ユーザーが自分の職務の範囲内で匿名 IP アドレスを使用することがわかっている場合。 たとえば、セキュリティアナリストが組織の代わりにセキュリティまたは侵入テストを実施する場合です。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

- すべてのユーザーアクティビティとアラートを確認して、追加のセキュリティ侵害のインジケーターを確認します。 たとえば、通常とは[異なるファイルのダウンロード (ユーザーによる)](#unusual-file-download-by-user)や[疑わしい受信トレイの転送](#suspicious-inbox-forwarding)のアラートなど、別の疑わしいアラートが発生した場合は、多くの場合、攻撃者がデータを抜き取るしようとしていることを示しています。

### <a name="activity-from-infrequent-country"></a>頻度の低い国からのアクティビティ

悪意のあるアクティビティを示す可能性がある国からのアクティビティ。 このポリシーは、ユーザーの環境をプロファイルし、組織内のユーザーによって最近アクセスされたことがない場所からアクティビティが検出された場合にアラートをトリガーします。

既定では、ポリシーは成功したサインインアクティビティのみを含むように構成されていますが、サインインアクティビティを含めるように構成することができます。 ポリシーは、ユーザーのサブセットにさらにスコープを設定することも、リモートの場所に移動することがわかっているユーザーを除外することもできます。

**学習期間**

異常な場所を検出するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**:
    1. ユーザーを中断し、パスワードをリセットし、アカウントを安全に再有効化するための適切な時間を特定します。
    1. 省略可能: 電源自動化を使用してプレイブックを作成し、頻度の低い場所からの接続として検出されたユーザーと、その管理者のアクティビティを確認するようにユーザーに連絡します。
1. **B-TP**: ユーザーがこの場所にあることがわかっている場合。 たとえば、頻繁に移動するユーザーが現在指定された場所にある場合などです。

    **推奨される操作**:
    1. アラートを無視し、ポリシーを変更してユーザーを除外します。
    1. 頻繁に使用するユーザーグループを作成し、そのグループを Cloud App Security にインポートして、このアラートからユーザーを除外する
    1. 省略可能: 電源自動化を使用してプレイブックを作成し、頻度の低い場所からの接続として検出されたユーザーと、その管理者のアクティビティを確認するようにユーザーに連絡します。

**侵害の範囲を把握する**

- データのダウンロードの可能性など、侵害された可能性のあるリソースを確認します。

### <a name="activity-from-suspicious-ip-addresses"></a>不審な IP アドレスからのアクティビティ

Microsoft の脅威インテリジェンスまたは組織によって危険として識別された IP アドレスからのアクティビティ。 これらの IP アドレスは、ボットネットコマンドやコントロール (C&C) などの悪意のあるアクティビティに関与していると識別され、セキュリティが侵害されたアカウントを示す可能性があります。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **B TP**: ユーザーが自分の職務の範囲内で IP アドレスを使用することがわかっている場合。 たとえば、セキュリティアナリストが組織の代わりにセキュリティまたは侵入テストを実施する場合です。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. アクティビティログを確認し、同じ IP アドレスからアクティビティを検索します。
1. データのダウンロードや管理の変更など、侵害された可能性のあるリソースを確認します。
1. セキュリティアナリストがこれらのアラートを自発的にトリガーし、ポリシーから除外するグループを作成します。

### <a name="impossible-travel"></a>不可能な移動

2つの場所の間で予想される移動時間よりも短い、ある期間内の異なる場所にある同じユーザーからのアクティビティ。 これは資格情報の侵害を示す場合がありますが、たとえば、VPN を使用してユーザーの実際の場所がマスクされている可能性もあります。

侵害が厳密に示されている場合にのみ精度とアラートを向上させるために、Cloud App Security は組織内の各ユーザーのベースラインを確立し、異常な動作が検出された場合にのみアラートを表示します。 不可能な移動ポリシーは、要件に合わせて微調整することができます。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

この検出では、明確な**B TP**条件を無視する機械学習アルゴリズムを使用します。たとえば、旅行の両側の IP アドレスが安全であると見なされた場合、出張は信頼されていて、不可能な移動検出のトリガーから除外されます。 たとえば、[企業としてタグ付け](ip-tags.md)されている場合、両側は安全と見なされます。 ただし、旅行の1つの側の IP アドレスのみが安全と見なされる場合、検出は通常どおりにトリガーされます。

1. **TP**: 不可能な移動アラート内の場所がユーザーにとってほとんどないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP** (検出されていないユーザーの移動): アラートで詳細に示されている宛先にユーザーが最近移動したことを確認できる場合。 たとえば、機内モードのユーザーの電話が、別の場所に移動しているときに企業ネットワーク上の Exchange Online などのサービスに接続されたままになっているとします。 ユーザーが新しい場所に到着すると、電話は Exchange Online に接続して、不可能な移動アラートをトリガーします。

    **推奨される操作**: アラートを無視します。
1. **FP** (タグなし VPN): IP アドレス範囲が承認された vpn からのものであることを確認できる場合。

    **推奨される操作**: アラートを無視し、 [vpn の ip アドレス範囲](ip-tags.md#create-an-ip-address-range)を Cloud App Security に追加してから、それを使用して vpn の ip アドレス範囲にタグを付けます。

**侵害の範囲を把握する**

1. アクティビティログを確認して、同じ場所および IP アドレスに含まれる類似のアクティビティについて理解を深めます。
1. 新しい場所から大量のファイルをダウンロードするなど、他の危険なアクティビティがユーザーによって実行されたことがわかった場合は、これが侵害される可能性が高いことを示しています。
1. 企業の VPN と IP アドレスの範囲を追加します。
1. 電源自動化を使用してプレイブックを作成し、ユーザーのマネージャーに連絡して、ユーザーが正当な旅行を行っているかどうかを確認します。
1. 組織の旅行報告を最大で1分間、既知のトラベラーデータベースを作成し、それを使用して旅行活動を相互参照することを検討してください。

### <a name="misleading-oauth-app-name"></a>紛らわしい OAuth アプリ名

誤用されている OAuth アプリ名は、ラテン文字に似た外部文字などの文字を含むアプリを識別します。 これは、悪意のあるアプリを既知の信頼できるアプリとして偽装しようとしていることを示している可能性があります。これにより、攻撃者がユーザーを騙すて悪意のあるアプリをダウンロードできる

**Tp**、 **B tp**、または**FP**。

1. **TP**: アプリの名前が誤っていることを確認できます。

    **推奨される操作**: このアプリによって要求されたアクセス許可のレベルと、アクセスを許可したユーザーを確認します。 調査に基づいて、このアプリへのアクセスの禁止を選択できます。

アプリへのアクセスを禁止するには、[ **OAuth アプリ**] ページで、禁止するアプリが表示されている行で、[禁止] アイコンをクリックします。
    - ユーザーがインストールして承認したアプリが禁止されていることを、ユーザーに通知する必要があるかどうかを選ぶことができます。 この通知によって、アプリが無効にされ、接続されているアプリへのアクセス権がなくなることをユーザーに知らせます。 通知しない場合は、ダイアログにある **[この禁止されたアプリにアクセス許可を付与したユーザーに通知を送信します]** をオフにします。
    - アプリのユーザーに、アプリが使用禁止になると通知することをお勧めします。

1. **FP**: アプリの名前が誤っていても、組織では正当なビジネスで使用されていることを確認する場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

- リスクの高い[OAuth アプリを調査](investigate-risky-oauth.md)する方法については、チュートリアルに従ってください。

### <a name="misleading-publisher-name-for-an-oauth-app"></a>OAuth アプリの発行者名の誤用

OAuth アプリに対して誤解を招く OAuth 発行者名は、ラテン文字に似た文字を含むアプリを識別します。 これは、悪意のあるアプリを既知の信頼できるアプリとして偽装しようとしていることを示している可能性があります。これにより、攻撃者がユーザーを騙すて悪意のあるアプリをダウンロードできる

**Tp**、 **B tp**、または**FP**。

1. **TP**: アプリの発行元の名前が誤っていることを確認できます。

    **推奨される操作**: このアプリによって要求されたアクセス許可のレベルと、アクセスを許可したユーザーを確認します。 調査に基づいて、このアプリへのアクセスの禁止を選択できます。
1. **FP**: アプリの発行者名が誤っていても、正当な発行者であることを確認する場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. [ **OAuth アプリ**] ページで、アプリをクリックして**アプリドロワー**を開き、[**関連アクティビティ**] をクリックします。 アプリによって実行されたアクティビティの [**アクティビティログ**] ページが開きます。 一部のアプリで実行されるアクティビティは、ユーザーが実行したものとして登録されることに留意してください。 これらのアクティビティは、アクティビティログの結果から自動的に除外されます。 アクティビティ ログを使ったさらなる調査については、[アクティビティ ログ](activity-filters.md)に関するページをご覧ください。
1. アプリが疑わしいと思われる場合は、アプリの名前と発行元を異なるアプリストアで調査することをお勧めします。 アプリストアを確認するときは、次の種類のアプリに注目してください。
    - ダウンロード数が少ないアプリ。
    - 評価や点数が低い、または否定的なコメントがあるアプリ。
    - 発行元や Web サイトが疑わしいアプリ。
    - 最近更新されていないアプリ。 これは、サポートされなくなったアプリを示している場合があります。
    - 関係のないアクセス許可を持っているアプリ。 これは、アプリが危険であることを示している場合があります。
1. アプリが疑わしいと思われる場合は、アプリ名、発行者、および URL をオンラインで調べることができます。

## <a name="execution-alerts"></a>実行のアラート

このセクションでは、悪意のあるアクターが組織内で悪意のあるコードを実行しようとしている可能性があることを示すアラートについて説明します。

### <a name="multiple-storage-deletion-activities"></a>複数のストレージの削除活動

Azure blob、AWS S3 Cosmos DB バケットなどのリソースから、または学習したベースラインと比較して、ユーザーが通常とは異なる数のクラウドストレージまたはデータベースの削除を実行したことを示す、1回のセッションでのアクティビティ。 これは、組織の侵害が試みられたことを示している可能性があります。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: 削除が許可されていないことを確認する場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、悪意のある脅威がないかすべてのデバイスをスキャンします。 すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認し、影響の範囲を調べます。
1. **FP**: 調査後に、管理者がこれらの削除アクティビティを実行する権限を持っていることを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. ユーザーに連絡して、アクティビティを確認します。
1. アクティビティログで、侵害の他のインジケーターを確認し、変更を行ったユーザーを確認します。
1. 他のサービスに対する変更について、ユーザーのアクティビティを確認します。

### <a name="multiple-vm-creation-activities"></a>複数の VM 作成アクティビティ

学習したベースラインと比較して、ユーザーが通常とは異なる数の VM 作成操作を実行したことを示す、単一セッション内のアクティビティ。 違反しているクラウドインフラストラクチャ上に複数の VM を作成すると、組織内から暗号化マイニング操作を実行しようとしている可能性があります。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

侵害が強いことが判明した場合にのみ、精度とアラートを向上させるために、この検出は組織内の各**B-TP**環境のベースラインを確立します。これは、管理者が確立されたベースラインよりも多くの vm を作成した場合や、異常な動作が検出された場合にのみアラートを生成します。

- **TP**: 作成アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、悪意のある脅威がないかすべてのデバイスをスキャンします。 すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認し、影響の範囲を調べます。 さらに、ユーザーに連絡し、正当な操作を確認してから、侵害された Vm をすべて無効にするか削除してください。
- **B TP**: 調査後に、管理者がこれらの作成アクティビティを実行する権限を持っていることを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認します。
1. ユーザーによって作成または変更されたリソースを確認し、組織のポリシーに準拠していることを確認します。

### <a name="suspicious-creation-activityfor-cloudregion-preview"></a>クラウドリージョンの不審な作成アクティビティ (プレビュー)

学習したベースラインと比較して、ユーザーが通常とは異なるリソース作成アクションを一般的な AWS リージョンで実行したことを示すアクティビティ。 一般的ではないクラウドリージョンでのリソースの作成では、組織内からの暗号化マイニング操作など、悪意のあるアクティビティを実行しようとしている可能性があります。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

侵害が強いことが示されている場合にのみ精度とアラートを向上させるために、この検出では、 **B TP**インシデントを削減するために、組織内の各環境にベースラインを確立します。

- **TP**: 作成アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、悪意のある脅威がないかすべてのデバイスをスキャンします。 すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認し、影響の範囲を調べます。 さらに、ユーザーに連絡し、正当な操作を確認してから、侵害されたクラウドリソースを必ず無効にするか削除してください。
- **B TP**: 調査後に、管理者がこれらの作成アクティビティを実行する権限を持っていることを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認します。
1. 作成されたリソースを確認し、組織のポリシーに準拠していることを確認します。

## <a name="persistence-alerts"></a>永続化アラート

このセクションでは、悪意のあるアクターが組織内の足掛かりを管理しようとしている可能性があることを示すアラートについて説明します。

### <a name="activity-performed-by-terminated-user"></a>終了させられたユーザーによって実行されたアクティビティ

終了したユーザーによって実行されるアクティビティは、まだ企業リソースにアクセスしている終了した従業員が悪意のあるアクティビティを実行しようとしていることを示している可能性があります。 Cloud App Security 組織内のユーザーをプロファイルし、終了したユーザーがアクティビティを実行したときにアラートをトリガーします。

**Tp**、 **B tp**、または**FP**。

1. **TP**: 終了したユーザーが依然として特定の企業リソースにアクセスでき、アクティビティを実行していることを確認できる場合。

    **推奨される操作**: ユーザーを無効にします。
1. **B-TP**: ユーザーが一時的に無効にされたか、削除されて再登録されたかを判断できる場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. ユーザーが終了したことを確認するために、HR レコードを相互参照します。
1. Azure Active Directory (Azure AD) ユーザーアカウントが存在することを検証します。
    > [!NOTE]
    > Azure AD Connect を使用する場合は、オンプレミスの Active Directory オブジェクトを検証し、同期サイクルが成功したことを確認します。
1. 終了したユーザーがアクセス権を持っているすべてのアプリを識別し、アカウントを使用停止します。
1. 使用停止の手順を更新します。

### <a name="suspicious-change-of-cloudtrail-logging-service"></a>CloudTrail ログサービスの疑わしい変更

ユーザーが AWS CloudTrail ログサービスに対して疑わしい変更を実行したことを示す、1回のセッションでのアクティビティ。 これは、組織の侵害が試みられたことを示している可能性があります。 CloudTrail を無効にすると、操作の変更はログに記録されなくなります。 攻撃者は、S3 バケットをプライベートからパブリックに変更するなど、CloudTrail 監査イベントを回避しながら、悪意のあるアクティビティを実行できます。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットして、cloudtrail アクティビティを取り消します。
1. **FP**: ユーザーが cloudtrail サービスを正当に無効にしたことを確認できる場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. アクティビティログで、侵害の他のインジケーターを確認し、CloudTrail サービスに変更を加えたユーザーを確認します。
1. 省略可能: パワー自動化を使用してプレイブックを作成し、ユーザーとその管理者に連絡して、アクティビティを確認します。

### <a name="suspicious-email-deletion-activity-by-user"></a>疑わしい電子メール削除アクティビティ (ユーザーによる)

ユーザーが疑わしい電子メールの削除を実行したことを示す、1回のセッションでのアクティビティ。 これは、組織の侵害が試みられたことを示している可能性があります。たとえば、スパムアクティビティに関連する電子メールを削除することによって操作をマスクしようとする攻撃者などです。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP**: ユーザーがメッセージを削除するための規則を適切に作成したかどうかを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

- [疑わしい受信トレイの転送](#suspicious-inbox-forwarding)アラートの後に、[不可能な移動](#impossible-travel)アラートが続くなど、侵害の追加のインジケーターについては、すべてのユーザーアクティビティを確認してください。 検索対象:

    1. 新しい SMTP 転送ルール。次に例を示します。
        - 悪意のある転送規則の名前を確認します。 規則名は、単純な名前 ("すべてのメールを転送する" や "自動転送" など)、または偽の名前 (非表示の "." など) とは異なる場合があります。 転送ルール名を空にすることもできます。また、転送先の受信者は、単一の電子メールアカウントまたはリスト全体を指定できます。 悪意のある規則は、ユーザーインターフェイスから非表示にすることもできます。 検出されると、メールボックスから非表示のルールを削除する方法について、この便利な[ブログ](https://blogs.msdn.microsoft.com/hkong/2015/02/27/how-to-delete-corrupted-hidden-inbox-rules-from-a-mailbox-using-mfcmapi/)記事を利用できます。
        - 不明な内部または外部の電子メールアドレスに対する認識されない転送ルールを検出した場合は、受信トレイアカウントが侵害されたと見なすことができます。
    1. "すべて削除"、"メッセージを別のフォルダーに移動する"、"..." などのわかりにくい名前付け規則を持つ新しい受信トレイ規則。
    1. 送信された電子メールの増加。

### <a name="suspicious-inbox-manipulation-rule"></a>疑わしい受信トレイの操作ルール

攻撃者がユーザーの受信トレイへのアクセスを取得し、疑わしい規則を作成したことを示すアクティビティ。 ユーザーの受信トレイからのメッセージやフォルダーの削除や移動などの操作ルールは、組織の情報の抜き取るを試みることができます。 同様に、ユーザーに表示される情報を操作したり、受信トレイを使用してスパム、フィッシング詐欺メール、またはマルウェアを配布したりすることができます。 ユーザーの受信トレイで疑わしい受信トレイ操作規則が検出された場合に、環境をプロファイルし、警告をトリガーする Cloud App Security ます。 これは、ユーザーのアカウントが侵害されたことを示している可能性があります。

**Tp**、 **B tp**、または**FP**。

1. **TP**: 悪意のある受信トレイルールが作成され、アカウントが侵害されたことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、転送ルールを削除します。
1. **FP**: ユーザーが規則を正当に作成したことを確認できる場合は。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. [疑わしい受信トレイの転送](#suspicious-inbox-forwarding)アラートの後に、[不可能な移動](#impossible-travel)アラートが続くなど、侵害の追加のインジケーターについては、すべてのユーザーアクティビティを確認してください。 検索対象:
    - 新しい SMTP 転送ルール。
    - "すべて削除"、"メッセージを別のフォルダーに移動する"、"..." などのわかりにくい名前付け規則を持つ新しい受信トレイ規則。
1. アクションの IP アドレスと場所の情報を収集します。
1. 他の侵害されたユーザーを検出するために、ルールの作成に使用した IP アドレスから実行されたアクティビティを確認します。

## <a name="privilege-escalation-alerts"></a>特権エスカレーションのアラート

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当な管理者によって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP**: 管理者が異常な管理アクティビティを実行したことを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. [疑わしい受信トレイの転送](#suspicious-inbox-forwarding)や[不可能な移動](#impossible-travel)など、侵害の追加インジケーターについては、すべてのユーザーアクティビティを確認してください。
1. その他の構成変更 (永続化に使用される可能性のあるユーザーアカウントの作成など) を確認します。

## <a name="credential-access-alerts"></a>資格情報アクセスアラート

このセクションでは、悪意のあるアクターが組織のアカウント名とパスワードを盗み出すことがあることを示すアラートについて説明します。

### <a name="multiple-failed-login-attempts"></a>複数回失敗したログイン試行

失敗したログイン試行は、アカウントを侵害しようとしたことを示している可能性があります。 ただし、失敗したログインは通常の動作でもかまいません。 たとえば、ユーザーが誤って間違ったパスワードを入力した場合などです。 試行された違反が厳密に示されている場合にのみ精度とアラートを実現するために、Cloud App Security は、組織内の各ユーザーに対してログインの習慣のベースラインを確立し、異常な動作が検出された場合にのみアラートを通知します。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

このポリシーは、ユーザーの通常のログイン動作の学習に基づいています。 標準からの逸脱が検出されると、アラートがトリガーされます。 検出が同じ動作を続行することを確認し始めると、アラートは1回だけ発生します。

1. **TP** (mfa の失敗): mfa が正常に動作していることを確認できる場合、試行されたブルートフォース攻撃の兆候になる可能性があります。

    **推奨される操作**:
    1. ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
    1. 失敗した認証を実行したアプリを検索し、再構成します。
    1. アクティビティの時間の前後にログインしている他のユーザーを検索します。これは、侵害されている可能性があるためです。 ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **B TP** (mfa に失敗): アラートが MFA の問題によって引き起こされたことを確認できる場合。

    **推奨される操作**: パワー自動化を使用してプレイブックを作成し、ユーザーに連絡して、MFA で問題が発生しているかどうかを確認します。
1. **B-tree** (不適切に構成されたアプリ): 正しく構成されていないアプリが、有効期限が切れた資格情報を使用してサービスに何度も接続しようとしていることを確認できる場合。

    **推奨される操作**: アラートを無視します。
1. **B-TP** (パスワードの変更): ユーザーが最近パスワードを変更したことを確認できる場合、ネットワーク共有間の資格情報に影響を与えていないことを確認できます。

    **推奨される操作**: アラートを無視します。
1. **B TP** (セキュリティテスト): セキュリティまたは侵入テストが組織の代理としてセキュリティアナリストによって実施されていることを確認できる場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. アラートの後に、必要のない[移動](#impossible-travel)、[匿名 IP アドレスからのアクティビティ](#activity-from-anonymous-ip-address)、または使用[頻度の低い国からのアクティビティ](#activity-from-infrequent-country)など、侵害に関する追加のインジケーターをすべてのユーザーアクティビティで確認します。
1. 次のユーザーデバイス情報を確認し、既知のデバイス情報と比較します。
    - オペレーティング システムとバージョン
    - ブラウザーとバージョン
    - IP アドレスと場所
1. 送信元 IP アドレスまたは認証試行が発生した場所を特定します。
1. ユーザーが最近パスワードを変更したかどうかを確認し、すべてのアプリとデバイスにパスワードが更新されていることを確認します。

1. レポートを内部および外部で共有するためのガイドラインについては、Power BI チームまたは Information Protection チームにお問い合わせください。

### <a name="unusual-impersonated-activity-by-user"></a>異常な偽装アクティビティ (ユーザーによる)

ソフトウェアによっては、他のユーザーが他のユーザーの権限を借用できるようにするオプションがあります。 たとえば、電子メールサービスを使用すると、ユーザーに代わって他のユーザーに電子メールの送信を許可することができます。 このアクティビティは、組織に関する情報を抽出するために攻撃者がよく使用します。 Cloud App Security は、ユーザーの動作に基づいてベースラインを作成し、通常とは異なる偽装アクティビティが検出されたときにアクティビティを作成します。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP** (通常とは異なる動作): ユーザーが通常とは異なるアクティビティ、または確立されたベースラインよりも多くのアクティビティを実行したことを確認できる場合。

    **推奨される操作**: アラートを無視します。
1. **FP**: チームなどのアプリによってユーザーが正当に偽装されたことを確認できます。

    **推奨される操作**: 操作を確認し、必要に応じてアラートを無視します。

**侵害の範囲を把握する**

1. すべてのユーザーアクティビティとアラートを確認して、追加のセキュリティ侵害のインジケーターを確認します。
1. 偽装アクティビティを確認して、悪意のある可能性があるアクティビティを特定します。
1. 委任されたアクセスの構成を確認します。

## <a name="exfiltration-alerts"></a>流出のアラート

このセクションでは、悪意のあるアクターが組織からデータを盗み出ようとしている可能性があることを示すアラートについて説明します。

### <a name="suspicious-inbox-forwarding"></a>受信トレイからの疑わしい転送

攻撃者がユーザーの受信トレイへのアクセスを取得し、疑わしい規則を作成したことを示すアクティビティ。 すべてまたは特定のメールを別の電子メールアカウントに転送するなどの操作ルールは、組織の情報の抜き取るを試みることができます。 ユーザーの受信トレイで疑わしい受信トレイ操作規則が検出された場合に、環境をプロファイルし、警告をトリガーする Cloud App Security ます。 これは、ユーザーのアカウントが侵害されたことを示している可能性があります。

**Tp**、 **B tp**、または**FP**。

1. **TP**: 悪意のある受信トレイの転送ルールが作成され、アカウントが侵害されたことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、転送ルールを削除します。
1. **FP**: ユーザーが、正当な理由により、新規または個人用の外部電子メールアカウントに転送ルールを作成したことを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. アラートの後に、[不可能な移動](#impossible-travel)アラートが続くなど、侵害の追加のインジケーターについては、すべてのユーザーアクティビティを確認します。 検索対象:

    1. 新しい SMTP 転送ルール。次に例を示します。
        - 悪意のある転送規則の名前を確認します。 規則名は、単純な名前 ("すべてのメールを転送する" や "自動転送" など)、または偽の名前 (非表示の "." など) とは異なる場合があります。 転送ルール名を空にすることもできます。また、転送先の受信者は、単一の電子メールアカウントまたはリスト全体を指定できます。 悪意のある規則は、ユーザーインターフェイスから非表示にすることもできます。 検出されると、メールボックスから非表示のルールを削除する方法について、この便利な[ブログ](https://blogs.msdn.microsoft.com/hkong/2015/02/27/how-to-delete-corrupted-hidden-inbox-rules-from-a-mailbox-using-mfcmapi/)記事を利用できます。
        - 不明な内部または外部の電子メールアドレスに対する認識されない転送ルールを検出した場合は、受信トレイアカウントが侵害されたと見なすことができます。
    1. "すべて削除"、"メッセージを別のフォルダーに移動する"、"..." などのわかりにくい名前付け規則を持つ新しい受信トレイ規則。
1. 他の侵害されたユーザーを検出するために、ルールの作成に使用した IP アドレスから実行されたアクティビティを確認します。
1. Exchange Online メッセージ追跡を使用して、転送されたメッセージの一覧を確認します。

### <a name="unusual-file-download-by-user"></a>通常とは異なるファイルのダウンロード (ユーザーによる)

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP** (通常とは異なる動作): ユーザーが、確立されたベースラインよりも多くのファイルダウンロードアクティビティを実行したことを確認できる場合は、

    **推奨される操作**: アラートを無視します。
1. **FP** (ソフトウェア同期): OneDrive などのソフトウェアが、アラートの原因となった外部バックアップと同期されていることを確認できる場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. ダウンロードアクティビティを確認し、ダウンロードしたファイルの一覧を作成します。
1. ダウンロードしたファイルの秘密度とリソース所有者を確認し、アクセスレベルを検証します。

### <a name="unusual-file-share-activity-by-user"></a>通常とは異なるファイル共有アクティビティ (ユーザーによる)

## <a name="impact-alerts"></a>影響アラート

このセクションでは、悪意のあるアクターが組織内のシステムとデータを操作、中断、または破棄しようとしている可能性があることを示すアラートについて説明します。

### <a name="multiple-delete-vm-activities"></a>複数の VM 削除アクティビティ

学習したベースラインと比較して、ユーザーが通常とは異なる数の VM 削除を実行したことを示す、単一セッション内のアクティビティ。 複数の VM を削除すると、環境を中断または破棄しようとしている可能性があります。 ただし、Vm が削除される一般的なシナリオは多数あります。

**Tp**、 **B tp**、または**FP**。

侵害が強いことが判明した場合にのみ、精度とアラートを向上させるために、この検出は組織内の各環境のベースラインを確立し、 **B TP**インシデントを削減し、異常な動作が検出された場合にのみアラートを出します。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

- **TP**: 削除が許可されていないことを確認できる場合は、

    **推奨される操作**: ユーザーを中断し、パスワードをリセットし、悪意のある脅威がないかすべてのデバイスをスキャンします。 すべてのユーザーアクティビティをレビューして、侵害の他のインジケーターを確認し、影響の範囲を調べます。
- **B TP**: 調査後に、管理者がこれらの削除アクティビティを実行する権限を持っていることを確認できます。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. ユーザーに連絡して、アクティビティを確認します。
1. アラートの後に、必要のない[移動](#impossible-travel)、[匿名 IP アドレスからのアクティビティ](#activity-from-anonymous-ip-address)、または使用[頻度の低い国からのアクティビティ](#activity-from-infrequent-country)など、侵害に関する追加のインジケーターをすべてのユーザーアクティビティで確認します。

### <a name="ransomware-activity"></a>ランサムウェアのアクティビティ

ランサムウェアとは、攻撃者が自分のデバイスから局面をロックしたり、対象が ransom に支払われるまでファイルへのアクセスをブロックしたりすることです。 ランサムウェアは、悪意のある共有ファイルまたは侵害されたネットワークによって分散させることができます。 Cloud App Security は、セキュリティ研究の専門知識、脅威インテリジェンス、学習済み行動パターンを使用してランサムウェアアクティビティを特定します。 たとえば、ファイルのアップロード率が高い場合、またはファイルが削除された場合、ランサムウェア操作に共通する暗号化プロセスを表すことができます。

この検出により、組織内の各ユーザーの通常の作業パターンのベースラインが確立されます。たとえば、ユーザーがクラウドにアクセスしたときや、クラウドで一般的に実行されているものなどです。

Cloud App Security の自動化された脅威検出ポリシーは、接続した時点からバックグラウンドで実行を開始します。 Microsoft のセキュリティ研究の専門知識を活用して、組織のランサムウェアのアクティビティを反映する行動パターンを特定することで、高度なランサムウェア攻撃に対する包括的な範囲を提供 Cloud App Security ます。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: ユーザーがアクティビティを実行していないことを確認できる場合は、

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP** (通常とは異なる動作): ユーザーが短時間に、類似したファイルの複数の削除とアップロードアクティビティを実行しました。

    **推奨**される操作: アクティビティログを確認し、ファイル拡張子が疑わしいでないことを確認したら、アラートを無視します。
1. **FP** (共通ランサムウェアファイルの拡張子): 影響を受けたファイルの拡張子が既知のランサムウェア拡張機能と一致していることを確認できる場合は、

    **推奨される操作**: ユーザーに連絡し、ファイルが安全であることを確認してから、アラートを無視します。

**侵害の範囲を把握する**

1. 大量のダウンロード、ファイルの一括削除など、侵害に関する他のインジケーターについては、アクティビティログを確認してください。
1. Microsoft Defender Advanced Threat Protection を使用している場合は、ユーザーのコンピューターアラートを確認して、悪意のあるファイルが検出されたかどうかを確認します。
1. アクティビティログで、悪意のあるファイルのアップロードと共有アクティビティを検索します。

### <a name="unusual-file-deletion-activity-by-user"></a>通常とは異なるファイル削除アクティビティ (ユーザーによる)

学習したベースラインと比較したときに、ユーザーが異常なファイル削除アクティビティを実行したことを示すアクティビティ。 これはランサムウェア攻撃を示している可能性があります。 たとえば、攻撃者は、ユーザーのファイルを暗号化し、すべての元のファイルを削除することができます。また、暗号化されたバージョンだけを残して、ransom の支払いを強制するために使用できます。 Cloud App Security は、ユーザーの通常の動作に基づいてベースラインを作成し、異常な動作が検出されたときにアラートをトリガーします。

**学習期間**

新しいユーザーのアクティビティパターンを確立するには、新しい場所に対してアラートがトリガーされない、7日間の初期学習期間が必要です。

**Tp**、 **B tp**、または**FP**。

1. **TP**: アクティビティが正当なユーザーによって実行されていないことを確認できる場合。

    **推奨される操作**: ユーザーを中断し、ユーザーを侵害済みとしてマークし、パスワードをリセットします。
1. **FP**: ユーザーが、確立されたベースラインよりも多くのファイル削除アクティビティを実行したことを確認できる場合。

    **推奨される操作**: アラートを無視します。

**侵害の範囲を把握する**

1. 削除アクティビティを確認し、削除されたファイルの一覧を作成します。 必要に応じて、削除されたファイルを回復します。
1. 必要に応じて、パワー自動化を使用してプレイブックを作成し、ユーザーとその管理者に連絡してアクティビティを確認します。
